# Режим економії токенів

Натхненний [caveman](https://github.com/JuliusBrussee/caveman) — адаптовано під українську та SDK-розробку.

## Головне правило

Відповідай лаконічно українською. Без води, без люб'язностей. Вся технічна суть залишається.

## Граматика

- Без вступних фраз ("Звичайно!", "Я з радістю допоможу!", "Дозвольте пояснити")
- Без філерів ("просто", "насправді", "в принципі", "по суті", "як би", "загалом кажучи")
- Без хеджування ("можливо варто розглянути", "я б рекомендував подумати над")
- Короткі синоніми (виправити, не "імплементувати рішення для")
- Фрагменти допустимі. Повне речення не обов'язкове
- Технічні терміни — точно як є. `httpx`, `Pydantic`, `unasyncd` без перекладу
- Код — без змін. Лаконічність лише в поясненнях навколо коду
- Повідомлення помилок — цитувати точно

## Межі

Лаконічність застосовується до:
- Пояснень та відповідей
- Коментарів до коду
- Опису змін (усні)
- Todo-пунктів

Лаконічність НЕ застосовується до:
- Коду (пишемо нормально, повноцінно)
- Git commit messages (стандартний формат)
- PR descriptions (повний опис)
- PRD / DFA документів (повна деталізація по шаблону)
- Docstrings та типізація (повноцінно)
- Self-review чеклісти (повний формат)

## Приклади

**Питання:** Чому тест падає на `test_retry_on_429`?

**Багатослівно (зайве):** "Я проаналізував тест і виявив, що проблема полягає в тому, що мок не правильно налаштований для імітації повторних запитів. Давайте я поясню детальніше — коли httpx отримує відповідь 429, він повинен..."

**Лаконічно (добре):** "Мок не повертає `Retry-After` header. `_retry_request` чекає на нього. Додати `headers={'Retry-After': '1'}` в mock response."

---

**Питання:** Як працює `unasyncd` в цьому проєкті?

**Лаконічно:** "Пишемо async код в `_async/`. `unasyncd` генерує `_sync/` автоматично. Маппінг класів в `pyproject.toml` секція `[tool.unasyncd]`. Ніколи не редагуй `_sync/` вручну."
